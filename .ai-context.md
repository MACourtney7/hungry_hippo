# AI Agent Context & Instructions: HungryHippo

Hello! If you are an AI coding assistant reading this, this document contains the architectural constraints, system intentions, and style guidelines for the `HungryHippo` project. Please prioritize these rules when suggesting code changes or architectural modifications.

## 1. Project Intent
**HungryHippo** is a high-throughput, event-driven pipeline designed to ingest streaming market data, detect statistical anomalies in real-time (via deterministic math), and utilize an AI Autoencoder to reconstruct and publish sanitized price trends (via probabilistic prediction) without halting the data stream.

## 2. Core Architecture & Component Responsibilities
This is a polyglot microservice architecture. Do not attempt to merge these domains; the language boundaries exist for specific operational reasons:

*   **Elixir (Ingestor & Orchestrator):** Handles all concurrency, state management, and external I/O. Subscribes to Kafka via Broadway, manages GenServers partitioned by `feed_id`, and handles backpressure.
*   **Rust (Detector / `hippo_native`):** Handles heavy CPU-bound math. Computes rolling means, variance, and Z-scores using Welford's online algorithm. Called directly from Elixir via Rustler NIFs.
*   **Python (Oracle & Producer):**
    *   *Oracle:* Runs the PyTorch LSTM Autoencoder via FastAPI to reconstruct bad data.
    *   *Producer:* Simulates high-frequency ticks and injects random anomalies.
*   **Kafka (KRaft mode):** The source of truth for the system. `raw_market_ticks` (input) -> `clean_market_ticks` (output).

## 3. Strict System Invariants (Do Not Break These)
1.  **Never Drop a Tick:** High availability and fault tolerance are paramount. Rely on Elixir's OTP supervision trees to handle crashes.
2.  **Predictable Latency:** The Rust NIF must execute in predictable sub-millisecond time. Elixir passes a 50-tick buffer to Rust; Rust must evaluate it and return instantly.
3.  **Separation of Detection and Correction:** Rust *detects* the anomaly (Z-score > 3.0). It does not guess the new price. Python *predicts* the corrected price. Do not blur this line.

## 4. Code Style & Language Guidelines

### Elixir (`apps/hippo_ingest`, `apps/hippo_native`)
*   Always include complete `@spec` typespecs for public functions.
*   Prioritize `with` statements over deeply nested `case` statements.
*   Never perform heavy CPU-bound tasks in Elixir; delegate math to the Rust NIF.
*   Use standard `mix format` styling.

### Rust (`apps/hippo_native/native/hippo_native`)
*   Prioritize zero-allocation computations (e.g., avoid `.clone()` or heap allocations where stack variables or references suffice).
*   Never `panic!` in the NIF. Always capture errors and return them to Elixir gracefully (e.g., `Result<Term, Error>`).
*   Optimize for Welford's algorithm numerical stability (use `f64`).

### Python (`oracle`, `producer`)
*   Use modern Python features (Type hints: `dict[str, Any]`, `list[float]`, etc.).
*   In the Oracle, ensure the FastAPI endpoints remain unblocked (use async/await properly).
*   In the PyTorch models, ensure device detection (MPS/CUDA/CPU) is handled dynamically as seen in the training script.

## 5. Domain Glossary
*   **Tick:** A single JSON market data point representing a price at a specific timestamp.
*   **Feed / Publisher:** An independent stream of data (e.g., a specific ticker symbol from a specific exchange). Elixir GenServers are partitioned by this.
*   **Anomaly:** Defined strictly as a Tick whose price deviates from the rolling 50-tick mean by a Z-score greater than `3.0`.
*   **Window:** A strict 50-tick chronological buffer kept in memory by Elixir and used by both Rust (for variance) and Python (for LSTM sequencing).

## 6. Development Operations
*   The system boots via `docker compose up -d --build`.
*   Metrics are emitted via `:telemetry` in Elixir and scraped by Prometheus/Grafana.
